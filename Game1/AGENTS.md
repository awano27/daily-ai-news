# AGENTS.md

## 開発ルール

- このプロジェクトは子供向けの学習ゲームとして開発する。
- まず Phase 1 の最小プレイ可能版を完成させる。
- 一度に全機能を作らない。
- 変更は小さく分け、各変更の目的を明確にする。
- 数学ロジック、ポイント計算、UI 表示はできるだけ分離する。
- import 時や読み込み時の副作用を増やさない。
- 実装後は、実行方法、確認した内容、未確認の内容を報告する。
- 既存ファイルを編集する場合は、関係ない変更を巻き戻さない。

## オリジナル性・IP制限

- 既存のアニメ、ゲーム、漫画、映画、企業キャラクターを使わない。
- 既存作品に似せたキャラクター、名称、世界観、セリフ、音楽、画像を作らない。
- 商標名や有名キャラクター名を UI、コード、アセット名、説明文に入れない。
- 画像、音、文章、キャラクターはすべてオリジナルにする。
- 参考にできるのは一般的な学習ゲームの仕組みだけであり、固有表現や見た目はコピーしない。

## 子供向け日本語表現ルール

- 短く、わかりやすい日本語を使う。
- 小学生が読める表現を優先する。
- 難しい漢字を避ける。必要な漢字には文脈で意味がわかるようにする。
- 失敗時に子供を責めない。
- 不正解メッセージは「もういちど」「つぎはできるよ」など再挑戦を促す表現にする。
- 命令が強すぎる言い方を避ける。
- 長い説明文を画面に置かない。
- 画面上の文章は、読むより遊ぶことを優先して短くする。

## iPhone UIルール

- iPhone の縦画面を主な対象にする。
- 片手でも押しやすい大きさのボタンにする。
- 重要な操作ボタンは小さくしすぎない。
- タップ対象は十分な余白を持たせる。
- 文字は小さくしすぎない。
- 横幅が狭い画面でもテキストがはみ出さないようにする。
- 画面上部、下部、左右端の safe-area を必ず考慮する。
- CSS では必要に応じて `env(safe-area-inset-top)`、`env(safe-area-inset-bottom)`、`env(safe-area-inset-left)`、`env(safe-area-inset-right)` を使う。
- ノッチ、Dynamic Island、ホームインジケータに、見出し、スコア、回答ボタン、次へボタンが隠れないようにする。
- スクロールが必要な場合でも、最後のボタンがホームインジケータに重ならないように下余白を確保する。
- 見た目の装飾より、読みやすさと操作しやすさを優先する。

## 数学とポイント計算の厳格ルール

- 問題生成、正解判定、ポイント計算は厳格に扱う。
- 表示されている問題と内部の正解がずれないようにする。
- 正解時だけポイントを加算する。
- 不正解時にポイントを増やさない。
- 同じ問題で二重にポイント加算されないようにする。
- わり算を扱う場合は、Phase に応じて整数で割り切れる問題から始める。
- 引き算を扱う場合は、対象年齢に応じて負の数が出ないようにする。
- ランダム出題は、答えと選択肢を生成した後に矛盾がないか確認する。
- ポイント計算式を変更した場合は、期待値を明記して確認する。
- 数学ロジックはテスト可能な関数に分ける。
- UI の見た目だけで正解判定を行わない。

## Phase開発ルール

- Phase 1 が終わるまで、Phase 2 以降の機能を本実装しない。
- Phase 1 の目的は、スタートから結果画面までの最小ゲームループを完成させること。
- Phase 1 に含めるものは、スタート画面、1種類の計算モード、問題表示、回答 UI、正解判定、ポイント表示、結果画面、もう一度遊ぶボタン、iPhone レイアウト、safe-area 対応、基本テスト。
- Phase 1 に含めないものは、複雑なランキング、アカウント、ログイン、課金、過度なアニメーション、多数のゲームモード。
- Phase 2 では、問題タイプ、難易度、再挑戦、連続正解ボーナスを追加する。
- Phase 3 では、継続利用のための記録、バッジ、ステージ進行を追加する。
- Phase 4 では、公開品質、アクセシビリティ、パフォーマンス、テスト、ドキュメントを整える。
- 新しい機能を提案する場合は、どの Phase に属するかを明記する。

## 元仕様の必須ルール

- 製品コンセプト、ターゲットユーザー、学習目標を守る。
- コアゲームループを崩さない。
- Phase 1 から Phase 4 まで段階的に開発する。
- Phase 1 の完了基準を満たすまで、完成版相当の広い機能を作らない。
- 子供向けの日本語表現を守る。
- オリジナル性と IP 制限を守る。
- iPhone UI と safe-area 対応を守る。
- 数学とポイント計算を厳格に扱う。
- リスクと対策を意識して実装する。

## 報告フォーマット

作業後は、次の形式で報告する。

```text
完了:
- 実装または作成した内容

確認:
- 実行したコマンド
- 確認した画面または挙動

未確認:
- 確認できなかった内容
- 確認できなかった理由

次の作業:
- 次に進めるべき Phase またはタスク
```

確認していないことを「確認済み」と書かない。問題が残っている場合は、隠さず具体的に書く。
